# 游戏代码开发规范与流程

> 目标：用明确的工程规范和开发流程，减少空引用、状态错乱、重复逻辑、资源缺失、热更事故、UI 映射异常、线上回归等不必要 bug。

## 1. 基本原则

1. 代码改动必须先明确业务意图、影响范围、入口、状态变化和回滚方式。
2. 游戏逻辑、UI 表现、资源配置、自动化工具、Editor 工具分层维护，禁止跨层随意写入。
3. 所有状态字段必须有明确状态机，不允许用临时字符串表达业务状态。
4. 所有配置、表格、资源引用必须可校验、可定位、可回滚。
5. 不允许把“当前能跑”当成完成标准，必须通过最小回归和关键路径验证。
6. 不允许绕过已有公共接口直接改底层数据，除非同步补齐索引、缓存、状态和校验。
7. 修 bug 时必须补充复发防护：校验、断言、测试、日志或工具检查至少一种。

## 2. 分层规范

### 2.1 业务逻辑层

- 只处理玩法规则、状态流转、数值结算、条件判断。
- 不直接依赖 UI 节点名、资源路径、图片名、Prefab 层级。
- 不在业务逻辑中写死活动 ID、道具 ID、时间戳、服务器开关。
- 任何状态切换必须走统一方法，例如 `SetState/ApplyResult/CommitReward`，禁止散落赋值。
- 有副作用的方法必须在命名上体现，例如 `ConsumeItem`、`GrantReward`、`SaveProgress`。

### 2.2 UI 层

- UI 只负责展示、输入、动画和转发用户意图。
- UI 点击不直接改背包、货币、任务、活动进度等核心数据。
- UI 打开时必须处理：数据未到、资源未加载、节点缺失、重复打开、关闭后回调。
- UI 刷新必须幂等，同一份数据重复刷新不能产生重复节点、重复事件、重复请求。
- 动态列表必须区分模板、实例、可见项、数据项，不能把第 N 个可见格子当成固定业务对象。

### 2.3 数据与配置层

- 配置表字段必须有默认值、类型、范围和枚举说明。
- 配置加载后必须做完整校验：主键重复、外键缺失、数值越界、空字符串、非法枚举。
- 客户端不可信任的字段必须由服务器裁决。
- 新增字段必须兼容旧数据：旧存档、旧缓存、旧活动配置不能崩溃。
- 删除字段前必须确认所有引用已迁移，并保留至少一个版本的兼容逻辑。

### 2.4 资源层

- 代码引用资源必须通过统一资源管理接口，禁止散落硬编码路径。
- Prefab、图片、音效、字体、特效、动画控制器必须有缺失兜底。
- 资源名变更必须同步引用检查。
- Unity `Assets` 下的 `.meta` 文件是资源 GUID 的来源，游戏工程内通常必须随资源提交。
- 自动化工具复制到 Unity 工程的 Editor 脚本必须只放在 `Assets/Editor/` 或约定目录，避免进入运行包。

### 2.5 自动化与工具层

- 自动化工具不得修改游戏业务代码。
- 工具生成的数据必须有 schema/version/source/generated_at。
- 工具写入正式映射、正式配置、正式索引前必须校验状态一致性。
- 临时运行文件、截图、bridge response、runtime snapshot 不进入正式提交。
- 任何自动生成目标不得覆盖已人工确认或已点击确认的正式目标。

## 3. 状态机规范

### 3.1 状态字段必须集中定义

每个状态字段必须有：

- 可选值列表
- 初始状态
- 允许的状态转移
- 终止状态
- 回退规则
- 显示文案
- 校验规则

示例：

```text
pending -> runtime_matched -> visual_confirmed -> click_confirmed -> case_verified
pending -> ignored
pending -> blocked
blocked -> pending
click_confirmed 不允许被普通刷新降级为 pending/runtime_matched/highlight_generated
```

### 3.2 禁止状态降级

以下状态一旦确认，不允许被刷新、推荐、生成目标、批量导入降级：

- `visual_confirmed`
- `click_confirmed`
- `case_verified`
- `collection_confirmed`
- `manual_confirmed`

如必须回退，必须显式传入 `allowStatusRegression=true`，并记录原因、时间和操作者。

### 3.3 状态必须和证据一致

如果证据显示：

- click 通过，则状态至少是 `click_confirmed`
- visual 已确认，则状态至少是 `visual_confirmed`
- runtime 已匹配，则状态至少是 `runtime_matched`

禁止出现正式证据已通过，但列表状态仍是 `pending`、`manual`、`highlight_generated`。

## 4. 命名规范

### 4.1 代码命名

- 类名表达对象，方法名表达动作，变量名表达含义。
- 布尔值使用 `is/has/can/should/allow` 开头。
- 异步接口必须在命名或返回类型上可识别。
- 不使用 `data/info/temp/new/old/obj` 这类无意义名称，除非作用域极小。

### 4.2 UI 与自动化命名

UI 自动化字段统一使用：

- `targetName`：中文业务名称
- `displayName`：页面内显示名称
- `semanticId`：稳定英文语义 ID
- `testId`：自动化执行 ID
- `reviewStatus`：审核状态
- `evidenceRef`：证据引用
- `formalRef`：正式映射引用

写入或修改 `semanticId/targetName/displayName` 时必须走统一语义流程：

1. 用户输入中文 `targetName`。
2. 如果中文词不在对象词典中，先翻译为英文对象词。
3. 按命名格式生成 `semanticId`。
4. 生成 `displayName`。
5. 将中文和英文对象词反补回对象词典。
6. 写入目标队列、formal、draft、evidence 时保持一致。

## 5. 空值与异常处理

1. 所有外部输入必须判空、判类型、判范围。
2. 回调中访问对象前必须确认对象仍有效。
3. UI 关闭、场景切换、资源卸载后，异步回调必须可取消或可忽略。
4. 捕获异常不能只吞掉，必须记录上下文：功能、玩家状态、配置 ID、资源路径、请求参数。
5. 对玩家可恢复的错误给兜底 UI，对不可恢复错误给明确日志和上报。

## 6. 数据写入规范

### 6.1 单一写入口

任何重要数据必须只有一个写入口：

- 玩家存档：统一存档服务
- 活动状态：统一活动状态服务
- 背包货币：统一资产服务
- UI 映射：统一 mapping store
- 目标队列：统一 target catalog

禁止多个模块直接写同一个 JSON、表格、缓存或存档字段。

### 6.2 写入必须同步派生数据

如果一个字段有派生数据，写入时必须同步：

- 主记录
- 索引
- 缓存
- legacy 兼容文件
- UI 展示状态
- 自动化证据状态

示例：正式映射写入后必须同步 formal、draft、evidence、index、target queue 的状态一致性。

## 7. 开发流程

### 7.1 开发前

1. 明确需求来源和验收标准。
2. 写清楚影响模块：业务、UI、资源、配置、协议、自动化。
3. 查现有实现，优先复用已有接口。
4. 列出风险点：状态、异步、资源、配置、跨服、热更、兼容。
5. 对高风险改动新建分支或 checkpoint commit。

### 7.2 开发中

1. 小步提交，小步验证。
2. 先改核心逻辑，再改 UI 展示，最后改工具和文档。
3. 每次新增状态、字段、配置，都同步补校验。
4. 每次修 bug，都补至少一个防复发机制。
5. 不在同一次改动里混入无关重构。

### 7.3 自测

最低自测清单：

- 新功能主流程可走通。
- 失败流程可恢复。
- 重复点击不会重复发奖、重复扣费、重复请求。
- 快速打开关闭 UI 不报错。
- 网络慢、数据为空、资源缺失时不崩。
- 老数据和默认配置可正常进入。
- 自动化校验或日志没有新增错误。

### 7.4 提交前

提交前必须执行：

1. `git status` 确认只包含本次相关文件。
2. `git diff` 复核核心逻辑。
3. 运行单元测试、脚本校验或最小回归。
4. 检查是否误提交截图、运行时快照、临时日志、`.bak` 文件。
5. 写清提交说明：改了什么、为什么改、怎么验证。

## 8. Code Review 规范

Reviewer 优先看这些问题：

- 状态是否可能被降级或错写。
- 是否有重复写入口。
- 是否绕过公共服务直接改数据。
- 异步回调是否处理对象销毁。
- 配置缺失是否会崩。
- 资源缺失是否有兜底。
- UI 重复刷新是否产生重复节点或重复事件。
- 是否影响旧存档、旧活动、旧客户端。
- 自动化目标和正式映射是否一致。
- 是否有必要的日志和验证手段。

Review 不以代码风格为主，优先拦截行为风险和线上事故。

## 9. 测试规范

### 9.1 必测场景

- 首次进入
- 重复进入
- 断线重连
- 资源未加载完成
- UI 快速打开关闭
- 多次点击
- 配置为空
- 配置非法
- 老存档升级
- 服务器返回异常
- 活动结束或条件不满足

### 9.2 游戏常见高风险点

- 奖励领取
- 货币扣除
- 付费购买
- 道具使用
- 引导流程
- 活动状态
- 红点系统
- 背包排序和堆叠
- 动态列表
- 跨天刷新
- 时间倒计时
- 战斗结算

这些点必须至少有主流程和重复操作验证。

## 10. UI 动态列表规范

动态列表必须区分：

- collection/template：集合规则
- item instance：可见实例
- data item：真实业务数据
- click area：点击区域
- visual/assert item：仅展示元素

禁止把以下元素混为一类：

- 资源栏加号集合
- 背包道具格子集合
- 道具数量文本
- 道具图标
- 选择状态图标
- 列表容器

动态集合写入 formal 前必须确认：

1. containerPath 指向集合容器。
2. itemPattern 能覆盖实例。
3. semanticPattern 能生成稳定 testId。
4. targetNamePattern 是业务中文。
5. 不和已确认实例重复。
6. 不生成无意义集合，例如 `TopRes/ResItem_{index}/add`。

## 11. 日志与监控

关键行为必须打结构化日志：

- 领取奖励
- 扣费
- 购买
- 道具使用
- 活动状态变化
- UI 打开失败
- 配置缺失
- 资源缺失
- 自动化点击失败
- 映射状态变更

日志必须包含：

- 玩家 ID 或测试会话 ID
- 功能名
- 操作名
- 关键配置 ID
- 前后状态
- 错误码
- 时间

## 12. 回滚规范

1. 代码改动必须可通过 Git 回滚。
2. 配置改动必须保留上一个可用版本。
3. 热更脚本必须有关闭开关。
4. 状态迁移必须可重复执行，不能执行第二次就破坏数据。
5. 线上修复必须记录修复原因、影响范围、验证结果。

## 13. 禁止事项

- 禁止直接改玩家资产字段。
- 禁止 UI 点击直接发奖或扣费。
- 禁止在业务代码里写死 UI 路径。
- 禁止新增状态但不加状态机。
- 禁止捕获异常后不记录。
- 禁止手工改 JSON 后不重建索引。
- 禁止把运行时快照当正式配置。
- 禁止把临时目标当正式映射。
- 禁止让自动生成覆盖人工确认结果。
- 禁止无验证地批量改名、批量改状态。

## 14. 开发完成定义

一项改动只有同时满足以下条件，才算完成：

- 功能符合需求。
- 主流程通过。
- 失败流程可恢复。
- 状态机无非法转移。
- 配置和资源校验通过。
- 自动化或手动最小回归通过。
- 没有无关文件和临时文件进入提交。
- 文档或注释补充了新的规则、字段或流程。
- 对本次 bug 有防复发机制。

## 15. 每日推荐流程

1. 拉取最新代码。
2. 查看当前分支和工作区状态。
3. 开发前确认需求和影响范围。
4. 小步实现。
5. 本地自测。
6. 跑校验脚本。
7. 提交或记录 checkpoint。
8. 推送远程。
9. 更新问题记录或进度文件。

## 16. 事故复盘模板

每个高风险 bug 修复后，必须记录：

```text
问题：
影响范围：
触发条件：
根因：
为什么测试没发现：
修复方式：
防复发措施：
验证结果：
是否需要补自动化：
是否需要补规范：
```

## 17. 最小检查清单

提交前快速过一遍：

- [ ] 是否有状态降级风险？
- [ ] 是否有重复写入口？
- [ ] 是否有空引用风险？
- [ ] 是否有异步回调访问已销毁对象？
- [ ] 是否有配置缺失兜底？
- [ ] 是否有资源缺失兜底？
- [ ] 是否会重复发奖、扣费、请求？
- [ ] 是否影响老数据？
- [ ] 是否需要重建索引或缓存？
- [ ] 是否跑过最小回归？
- [ ] 是否有防复发校验？
