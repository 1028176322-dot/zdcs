# AutoSmoke IDE 功能验收标准

## 1. 目标与范围

这份文档用于验收 AutoSmoke IDE 在以下维度的完整性和可交付能力：

- 环境与项目接入能力
- 用例执行能力（按模板）
- 自动探索能力（页面、弹窗、主城、地图）
- 异常检测与判定能力
- 页面关系图与结果报告能力
- 跨路径/跨设备可复现运行能力

验收目标是判断“是否可在真实项目上稳定闭环运行”，而不仅是单点功能可用。

验收定义：

- **通过**：功能在规定条件下稳定达到标准且可重复执行。
- **基本通过**：功能可运行，但存在已知边界条件限制，需要在 Release Notes 记录。
- **不通过**：核心行为未达标，必须修复后才能进入下一阶段。

## 2. 验收范围

- 目标平台：Windows 机器
- 运行对象：Unity 游戏项目（Unity Editor 或 Windows 包）
- 用例来源：`用例模板.xlsx`（兼容导入）
- 输出目标：报告、日志、截图、页面图谱、Bug 列表

## 3. 功能验收总原则

1. 每个模块必须有明确入口、输出和失败回退。
2. 每个模块都要有“成功/失败/异常”三态记录。
3. 任何自动化判断必须保留可复现证据。
4. 验收结果必须有可追溯文件路径（`.autosmoke/runs/<run_id>/...`）。
5. 同一场景同条件重跑结果偏差应可解释，且不应出现随机失败。

## 4. 分模块验收标准

## 4.1 环境配置与项目接入

### 4.1.1 项目路径识别

- 验收标准
  - 输入任意有效 Unity 项目路径后，IDE 能识别并打开项目。
  - 能发现 `Assets/`、`ProjectSettings/ProjectVersion.txt`、`Packages`。
- 通过条件
  - 显示“项目可用”状态。
  - 可写入 `.autosmoke/` 目录。
- 失败条件
  - 路径无效或非 Unity 工程时，明确报 `PROJECT_INVALID` 并阻断后续步骤。
- 验收输出
  - `project_info.json`
  - 项目检测日志

### 4.1.2 Poco SDK 检查与接入

- 验收标准
  - 自动检测 Poco 是否已存在。
  - 未接入时可一键完成接入（或提示明确缺失文件）。
- 通过条件
  - `poco.connect()` 前置检查通过。
  - `poco.dump()` 至少返回根节点。
- 失败条件
  - 导入未成功且未提示具体缺失项（如关键脚本、asmdef、编译中）。
- 验收输出
  - `setup_report.json`
  - 插件导入日志

### 4.1.3 配置文件生成

- 验收标准
  - 按项目生成 `.autosmoke/config.json`，支持 `${PROJECT_PATH}`。
  - 旧路径模板可在迁移时自动替换。
- 通过条件
  - 启动后能读取配置并恢复会话。
- 失败条件
  - 配置不可读取或关键字段缺失导致运行中断且无自动修复提示。

## 4.2 连接与运行前自检

### 4.2.1 设备连接

- 验收标准
  - 支持 Windows Editor / Android 至少一种可运行路径。
- 通过条件
  - 连接状态显示 `connected`。
  - 连接丢失后 30 秒内有重连提示或告警。
- 失败条件
  - 长时间连接无状态更新，任务继续进行。

### 4.2.2 运行前健康检查

- 验收标准
  - 完成至少4项基础自检：设备、Poco、截图、日志监听。
- 通过条件
  - 4 项结果至少都为绿，或有明确白名单豁免说明。
- 失败条件
  - 截图/日志采样失败但系统误判可继续执行。

### 4.2.3 场景状态读取

- 验收标准
 - 若接入 SceneStateExporter，运行前后每步都可获取场景摘要。
- 通过条件
  - 当前场景名可解析。
- 失败条件
  - 场景导出缺失未在异常中心提示。

## 4.3 用例模板兼容性验收

### 4.3.1 Excel 模板读取

- 验收标准
  - 读取 `E:/zdcs/参考资料/用例模板.xlsx` 风格的表头与字段。
  - 识别第 13 行表头、第 14 行起始数据。
- 通过条件
  - 能成功导入至少 3 条样例用例。
- 失败条件
  - 表头识别错误或字段错位导致全量导入失败。
- 验收输出
  - `import_result.json`
  - 导入失败行清单

### 4.3.2 字段映射规则

- 验收标准
  - 支持 B~L 列映射到内部模型。
  - 非空行至少包含 `CaseID`，否则跳过并记录。
- 通过条件
  - 生成内部执行结构并显示可执行步骤数。
- 失败条件
  - 无法保留 `CaseID` 与 `测试结果` 回写能力。

### 4.3.3 自然语言解析

- 验收标准
  - 对 `操作步骤`、`预期结果` 至少能解析核心动作模板（点击、输入、跳转、等待、断言）。
- 通过条件
  - 已有规则可解析 80%以上示例文本。
- 失败条件
  - 大量规则命中不到后不提示人工映射。

### 4.3.4 结果回填

- 验收标准
  - 执行后回填 `测试结果` 和 `BUG链接`。
- 通过条件
  - 最少一条 PASS 与一条 FAIL 的用例能正确回填。
- 失败条件
  - 回填后文件损坏或字段乱码。

## 4.4 用例执行引擎验收

### 4.4.1 单步执行

- 验收标准
  - 前置动作->执行->断言闭环可重复。
- 通过条件
  - 每步都有前置/后置证据（截图或日志）保存。
  - 单步超时有明确异常原因。
- 失败条件
  - 步骤失败后无法继续展示原因与日志。

### 4.4.2 多步用例

- 验收标准
  - 支持连续步骤顺序执行并按失败短路。
- 通过条件
  - 步骤 1 通过后，步骤 2 正常执行。
  - 步骤失败后该用例终止，后续用例可继续。
- 失败条件
  - 失败用例阻塞队列。

### 4.4.3 执行结果判定

- 验收标准
  - 实现 `PASS/FAIL/BUG/ERROR/WARNING/SMOKE_PASS/SKIPPED`。
- 通过条件
  - 至少3种判定能在真实运行中出现并正确分类。
- 失败条件
  - 仅用通用“成功/失败”输出，不区分异常类型。

### 4.4.4 暂停与停止

- 验收标准
  - 支持暂停、继续、手动停止。
- 通过条件
  - 停止后生成中断报告。
- 失败条件
  - 任务停不下来或状态机损坏无法恢复。

## 4.5 自动探索验收

### 4.5.1 页面遍历

- 验收标准
  - BFS/队列遍历可覆盖新页面与弹窗分支。
- 通过条件
  - 产生 `page_graph.json`，节点 > 0。
- 失败条件
  - 同一页面重复重复访问无去重。

### 4.5.2 弹窗遍历

- 验收标准
  - 识别新弹窗并尝试关闭。
- 通过条件
  - 关闭策略顺序一致（关闭按钮/取消/返回/遮罩）。
- 失败条件
  - 弹窗阻断后无 BLOCKED 标记和恢复逻辑。

### 4.5.3 探索安全性

- 验收标准
  - 黑名单生效，危险动作不执行。
- 通过条件
  - 命中危险标签时记录 `BLOCKED` 并不执行动作。
- 失败条件
  - 风险动作被执行。

## 4.6 主城场景能力验收

### 4.6.1 主城对象识别

- 验收标准
  - `MainCity` 场景下导出建筑与可交互对象。
- 通过条件
  - 每步记录对象 ID、名称、坐标、状态。
- 失败条件
  - 仅有 UI 树、无场景对象时仍显示“主城扫描通过”。

### 4.6.2 建筑点击与弹窗

- 验收标准
  - 点击建筑能触发对应弹窗或状态变化。
- 通过条件
  - 识别建筑 -> 点击 -> 跳转/弹窗 -> 截图变化。
- 失败条件
  - 所有建筑点击皆无任何差异无异常标记。

## 4.7 大地图场景验收

### 4.7.1 缩放与拖拽

- 验收标准
  - 支持 `zoom_in/zoom_out/pan` 操作。
- 通过条件
  - 缩放等级变化可读取且对对象可见性有影响。
- 失败条件
  - 操作成功但状态未更新，或对象列表不变。

### 4.7.2 网格扫描

- 验收标准
  - 能按照配置扫描多个网格点并聚合对象。
- 通过条件
  - 记录 `grid_id`、中心坐标、zoom、对象数。
- 失败条件
  - 仅扫描单屏，无法覆盖范围模式。

### 4.7.3 对象点击链路

- 验收标准
  - 对对象执行 click 后有明确 post-action 差异。
- 通过条件
  - 至少 1 种对象类型可触发弹窗/页面跳转。
- 失败条件
  - 所有对象点击后无反应且无异常。

## 4.8 异常检测验收

### 4.8.1 崩溃检测

- 验收标准
  - 检测到连接断开、进程退出、致命错误。
- 通过条件
  - 崩溃可立即标记 `BUG` 并写入复现路径。
- 失败条件
- 崩溃后仍显示 PASS。

### 4.8.2 卡死检测

- 验收标准
- SOFT/HARD 冻结能区分。
- 通过条件
  - 超时后给出具体步骤与等待时长。
- 失败条件
  - 长时间无反应但持续刷日志，系统未停更改判定。

### 4.8.3 空页面检测

- 验收标准
  - 检测到纯黑/纯白/纯色、可交互对象为空。
- 通过条件
  - 生成 WARNING 或 BUG，并附截图和 ui_tree。
- 失败条件
  - 空页面被误判为正常通过。

### 4.8.4 Missing Reference / NullReference

- 验收标准
  - 错误日志可自动匹配并升级为 `BUG`。
- 通过条件
  - 至少 1 次触发时产生 `BUG-xxx` 条目。
- 失败条件
  - 仅显示执行失败，不归类为异常类问题。

### 4.8.5 日志采样质量

- 验收标准
  - 每步记录“本步新增日志”。
- 通过条件
  - 日志片段可回放到步骤复现。
- 失败条件
  - 日志为空或全量文件不切片，无法定位步骤。

## 4.9 页面关系图验收

### 4.9.1 节点与边

- 验收标准
  - 每次页面/弹窗/对象导航都形成图边。
- 通过条件
  - `page_graph.json` 包含 nodes / edges 两类。
- 失败条件
  - 只有节点无边，或边缺失动作信息。

### 4.9.2 可视化展示

- 验收标准
  - IDE 展示关系图并可展开异常节点。
- 通过条件
  - 支持按场景筛选、按时间排序。
- 失败条件
  - 图不渲染或缺失关键路径。

## 4.10 报告验收

### 4.10.1 总体报告

- 验收标准
  - 输出 `index.html` 和 `summary.json`。
- 通过条件
  - 报告含总计、通过、失败、BUG、警告、运行时长。
- 失败条件
  - 关键指标为空或格式错误。

### 4.10.2 用例与步骤明细

- 验收标准
  - `cases.json` 覆盖所有执行用例。
- 通过条件
  - 每条失败/异常带证据链接（截图、日志、树文件）。
- 失败条件
  - 失败用例缺失复现证据。

### 4.10.3 Bug 列表

- 验收标准
  - `bugs.json` 自动聚合，按严重级别排序。
- 通过条件
  - 包含 repro_path、日志、证据路径。
- 失败条件
  - BUG 与执行结果不同步。

### 4.10.4 导出与归档

- 验收标准
  - 每次运行独立目录，不互相覆盖。
- 通过条件
  - `screenshots/ui_trees/scene_states/logs` 全部在本次 run 目录。
- 失败条件
  - 不同 run 写入同一文件导致覆盖。

## 5. 功能完整度评分模型

定义三档状态：

- A档（可发布）：验收项达到 95% 以上，且 P1/P0 级缺陷为 0。
- B档（可内测）：验收项达到 85%~94%，允许已知问题清单与临时规则。
- C档（不可发布）：验收项低于 85%，或存在关键缺陷阻断闭环。

建议评分公式：

```text
完整度 = (通过项数 / 可验收项数) * 100%
```

权重建议：

- 用例执行与报告：35%
- 连接与稳定性：20%
- 自动探索：20%
- 异常检测：15%
- 适配与迁移：10%

最低通过线：

- 每个模块单项不得低于 70%。
- 核心功能（用例执行+报告）不得低于 80%。

## 6. 阶段性验收检查项

### 阶段一：基础闭环

- 连接、dump、截图、用例执行（简单断言）通过。
- 目标通过率：>=80%
- Bug 检测：仅 log_no_error + 基础崩溃

### 阶段二：异常与稳定性

- 崩溃、卡死、空页面、Missing Reference 检测通过。
- 自动回填报告可读取异常路径。
- 目标通过率：>=85%

### 阶段三：探索与关系图

- 页面遍历 + 弹窗遍历 + 关系图 JSON/HTML 通过。
- 目标通过率：>=85%

### 阶段四：场景增强

- 主城对象导出和大地图扫描通过。
- 缩放拖拽扫描策略可执行。
- 目标通过率：>=80%

### 阶段五：跨环境可迁移

- 同项目异地路径下可重复执行。
- 配置迁移不依赖固定路径。
- 目标通过率：>=90%

## 7. 交付验收清单（Checklist）

- 环境配置向导可运行
- 连接并完成自检
- 用例模板导入成功
- 用例执行有日志与截图
- 失败与 BUG 自动分类
- 自动探索可生成页面关系图
- 主城与大地图扫描可执行
- 报告含复现路径与证据
- 跨电脑路径迁移后可再次运行

每项达到“是”后方可进入下一阶段。

- 失败回退说明（至少一种可恢复或明确阻断）

### 矩阵评分规则

- 每个“必须项”记为 1 分，缺失 1 个减少 1 分。
- “可选项”通过加 0.5 分，未支持不作为总分阻塞项，但影响完整度档位。
- 模块最低门槛 70%，并要求该模块全部“必须项”至少 90%。
- 核心链路门槛（自检→执行→报告）不低于 85%。

### 完整度达标建议

1. P0/P1 缺陷数必须为 0，且有明确回归修复人和时间。  
2. 若出现“可复现性问题”（同一脚本两次执行结果差异）超过 3% ，判定 B 或以下。  
3. 任一关键链路步骤出现 2 次以上阻断（不是偶发网络/环境噪音）应触发“返工”状态。
